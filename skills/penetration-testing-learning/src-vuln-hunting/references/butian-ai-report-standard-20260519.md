# 补天AI生成漏洞报告提交规范 (2026-05-19)

> 补天漏洞响应平台联合30+SRC共同发布《AI生成漏洞报告提交规范》

## 背景

随着大模型能力日益增强，利用AI工具辅助或自动化挖掘漏洞已成为安全研究的新趋势。近期，平台收到了一些由AI协助发现的高质量漏洞报告，但同时也面临大量未经人工验证的无效报告，甚至是被AI"幻觉"误导而产生的误报。

## 鼓励AI使用

补天漏洞响应平台积极鼓励白帽子利用AI工具辅助或自动化挖掘漏洞，提升发现漏洞的效率，挖掘更多潜在风险

## 严格人工验证

对于AI辅助或自动化挖掘产出的漏洞报告，提交前务必进行人工验证。请确保您已完成对报告真实性与危害性的评估和复现，并在报告中详细提供人工验证结果，包括但不限于：
- 漏洞危害说明
- 详细复现步骤
- 完整的POC
- 关键步骤及结果截图

## 无效报告处理

对于直接由AI生成、未经人工复现或未提供有效验证结果截图的报告，平台将直接予以驳回，且不提供驳回原因说明

## 违规报告处置

- 累计提交超过3个AI生成报告且未经上报者人工验证报告有效性 → 平台提醒
- 累计提交超过5个AI生成报告且未经上报者人工验证报告有效性 → 扣除信誉值，情节严重者封号处理

## 对SRC测试的影响

1. **AI挖掘被鼓励** — 用AI辅助发现漏洞是合规的
2. **人工验证是硬性要求** — 没有验证的报告直接驳回，不给原因
3. **截图是核心证据** — 关键步骤必须有截图证明人工验证过
4. **批量AI报告会被识别** — 不要一次性提交大量相似格式的AI报告
5. **处罚机制严格** — 5个未验证报告就可能封号

## 报告提交检查清单

提交前确认：
- [ ] 漏洞是否仍然存在（手动执行curl/访问URL）
- [ ] 是否有完整的复现步骤（分步骤，每步有命令+响应）
- [ ] 是否有可直接执行的POC（curl命令或脚本）
- [ ] 是否有关键步骤截图（证明人工验证过）
- [ ] 是否有实际的危害证明（非理论推断）
- [ ] 报告格式是否符合平台要求（纯文本/HTML）

## 与企业SRC报告要求的对比

| 要素 | 补天公益SRC | 企业SRC(漏洞盒子) |
|------|------------|-------------------|
| 人工验证 | 必须(有截图) | 必须(有完整数据包) |
| POC | 完整可执行 | 完整可执行 |
| 截图 | 必须 | 必须 |
| HTTP数据包 | 建议提供 | 必须完整 |
| 处罚 | 3次提醒/5次封号 | 审核退回 |

## 实战: 批量报告被驳回后的修复工作流 (2026-05-19)

当多份报告同时被驳回时：

1. **先验证漏洞是否仍存在** — 用之前的POC命令直接调用，不要假设还在
2. **检查前端是否修复但密钥未轮换** — JS中移除了硬编码≠密钥被轮换，必须验证已泄露密钥是否仍可用
3. **重写报告并标注截图位置** — 在报告中用【截图位置N】标明用户需要截图的地方
4. **提供可直接复制的单行curl命令** — 多行命令在zsh/bash终端会报错
5. **确认用户能复现后再提交** — 不要让用户"试试看"，要确认能拿到数据
