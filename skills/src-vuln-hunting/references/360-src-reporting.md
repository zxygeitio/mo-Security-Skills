# 360 SRC / 众测报告与验证要点

## 报告字段约束

360 提交页常见字段：
- 所属域名：只填域名或 IP，例如 `pms.zgcbank.com`。
- 漏洞标题：厂商 + 系统/功能点 + 漏洞类型，简短清晰。
- 漏洞类型：若没有 CORS 选项，不要硬套；CORS 配置类可选“其他”，未授权/注册链校验类优先选“权限控制缺失”，短信/验证码滥用可选“逻辑漏洞”。
- 风险级别：低危/中危/高危按实际危害保守选择。
- 漏洞简述：平台限制约 200 字以内，不放 URL/Payload/长命令，只写概况和危害。
- 漏洞 URL：漏洞点所在 URL。
- 复现步骤：平台限制约 5000 字以内；不要放长 HTML/完整响应头；只保留关键命令、关键返回字段、截图位置。
- 修复方案：短条目。
- 附件：按截图位置编号。

## 报告压缩模板

漏洞简述（<=200字）：
`[厂商][系统/功能]存在[漏洞类型]。[攻击者条件]即可[触发/读取/操作]，导致[实际危害]。`

复现步骤（<=5000字）：
1、访问公开页面获取必要参数（SESSION/CSRF/接口路径）。【截图位置1】
2、执行最小 POC 命令 1，贴关键 JSON/状态码。【截图位置2】
3、执行最小 POC 命令 2（如需证明多接口/差异），贴关键 JSON/状态码。【截图位置3】
4、结论：说明无需登录/越权/绕过的核心证据。

避免：
- 重复解释漏洞原理。
- 贴完整 HTML 页面。
- 把修复建议、影响分析混进复现步骤。
- 给出 8000 字以上的长复现。

## 证据截图原则

每张截图都必须同时包含：
- 命令或请求 URL；
- 关键参数；
- 关键响应字段；
- 能证明当前步骤的结果。

推荐截图结构：
- 截图1：公开页面提取 CSRF/接口路径/必要字段。
- 截图2：未登录或越权请求返回业务结果。
- 截图3：对照请求或第二个接口返回差异/成功。
- 截图4：影响证明（例如发送间隔、数据回显、越权状态）。

## 已提交漏洞去重纪律

在同一 SRC 项目中，后续挖掘必须维护“已提交漏洞集合”。同一根因的增强描述不能当作新漏洞。例如：
- “注册链未授权校验接口”与“同一组注册链接口 CSRF 头校验不严”若依赖相同入口、相同接口、相同危害，应视为同一类问题，不要重复提交。
- 新报告必须满足至少一个条件：不同资产、不同业务链、不同权限边界、不同影响数据、或可证明更高危害（如从校验接口提升到任意重置密码/越权读写/短信轰炸）。

## 命令书写

给用户的 curl 命令必须是单行，避免复制到 zsh/bash 后换行导致命令变形。需要使用变量时，先单独设置变量，再给单行 curl。

当用户要求“报告直接输出”“命令一条条执行”“中间不要有空行”时：
- 不要使用 heredoc（`bash <<'EOF'`）作为复现命令格式；很多终端复制时会因为缩进、自动换行或 `EOF` 不顶格进入 `heredoc>`。
- 每个复现动作拆成单独一行命令：先 `curl -D ... -o ... URL`，再 `head -1 ...`，再 `wc -c ...`，再 `grep ... file | sort -u`。
- 报告复现步骤内不要插入空行；每条命令保持完整单行，不使用行尾反斜杠续行。
- 如果必须解释 heredoc 卡住问题，告诉用户 `Ctrl+C` 退出，并说明 `EOF` 必须顶格且单独一行。

常见提取 SESSION 错误：权限控制缺失导致越权读写；
- 认证绕过/密码重置缺陷；
- 可证明的短信/邮箱轰炸或资源消耗；
- 真实业务链注册/登录/找回密码逻辑缺陷。

谨慎提交：
- 仅“接口可调用但返回无敏感内容”的低价值校验；
- 仅错误配置但无实际影响；
- 同根因重复问题；
- 需要猜测危害、没有人工复现证据的问题。
