# BZUU / 亳州学院深度复测负向结论模式（2026-06-05）

适用场景：教育 SRC 目标已有历史候选/负向报告，本轮要求“深度挖掘实质漏洞”。本参考用于避免把历史边界问题、可上传但不可利用、公开配置泄露重复包装成新洞。

## 必做顺序

1. 先读取历史报告并去重：确认同根因是否已经提交/负向复测过。
2. 对候选漏洞做 Evidence → Hypothesis → Validation：
   - 能否未授权读取敏感业务数据？
   - 能否越权读取/修改其他用户数据？
   - 能否账号接管、认证绕过、RCE、SQLi、任意文件下载？
   - 是否只有配置、默认页面、公开学校元数据、登录初始化信息？
3. 必须有对照组：敏感接口如果返回 401/Token 失效，不要把公开配置接口的 CORS/未授权扩大成“可读取敏感数据”。
4. 输出时明确：新洞可提交 / 仅旧洞补证 / 不建议提交。

## BZUU 本轮关键判定

### go-fastdfs 未授权上传

现象：`https://oshall.bzuu.edu.cn/fileServer/upload` 可上传安全 marker 文件并返回可访问路径。

验证重点：下载响应头。

判定逻辑：
- 如果下载 `Content-Type` 被强制为 `application/octet-stream` 或 `text/plain`，且没有脚本执行、HTML/JS 渲染、存储型 XSS、敏感文件覆盖、路径穿越，则只能作为“未授权上传/文件存储配置不当”的旧问题补证。
- 不要包装成“可执行文件上传高危”。
- 若历史已有同根因，除非出现新的可利用危害，否则不新建报告。

### getAppConfig 配置接口

现象：`/zhxyApi/rhpt-applets/applets/synthesis/query/getAppConfig` 未授权返回初始密码规则、CAS 地址，并存在 CORS Origin 反射 + credentials。

判定逻辑：
- 该类问题最多是中危边界，必须保守描述。
- 如果学生通讯录、教师通讯录、成绩、工资、数据源、附件等接口均返回 `401 Token失效`，不能声称账号接管或敏感数据读取。
- 若历史已提交/复测，不建议重复提交。

### 新资产公开元数据

第二课堂、CAS、JYXT、HCM、Lib、Mail、OA/WebVPN 等系统如果只暴露：学校名称、logo、登录页初始化 XML、AppCode、404/302/空体、默认页，不构成可提交实质漏洞。

## 推荐最终输出措辞

“本轮未验证到新的、建议独立提交的实质性高危/中高危漏洞。已复现的 go-fastdfs 上传和 getAppConfig 配置泄露属于历史同根因/中危边界；敏感业务接口均需 Token，未证明数据读取、账号接管、越权、RCE、SQLi 或任意文件下载。因此不建议提交新报告。”

## 报告落盘注意

如果用 Python 生成中文报告，避免在旧解释器/受限沙箱中把复杂中文直接写进 f-string 表达式位置；优先使用普通三引号字符串、`.format()` 或字符串拼接，写入后必须 read back 验证。不要把一次性 SyntaxError 记录成工具不可用结论，记录的是“报告生成要读回验证”的流程。
