# 360 SRC / 中关村银行漏洞被忽略后的深挖收敛记录 (2026-05-20)

适用技能: `src-vuln-hunting`

## 背景
用户反馈此前提交的异常脚本、Tomcat版本、health页面、PMS弱信息泄露等均被360忽略。后续挖掘必须停止用低危/配置/信息泄露凑数，改为只保留可证明实质危害的链路。

## 本轮已验证但不应再作为主要提交的方向

### 1. 边缘资产异常脚本 / Tomcat / health.html
- `app/wx/www` 异常脚本、Tomcat错误页版本、health页面暴露都容易被平台按低危/无实质危害忽略。
- 除非能证明页面被真实篡改、影响真实用户、可执行恶意脚本或造成账号/数据风险，否则不要继续作为主报告。

### 2. PMS找回密码链
- `/pms/ananymous/zzqx/zhmm/confirmAccount`
- `/pms/ananymous/zzqx/zhmm/valiYzm`
- 能进入验证步骤并出现手机号掩码，但随机账号也会返回掩码。
- 不要再写“账号枚举”或“账号接管”。只能弱表述为找回密码流程权限控制缺失/手机号掩码泄露，价值偏低。

### 3. PMS注册字段校验
- `checkDlh`
- `checkZtmc`
- `checkZtdmByZc`
- `admin` 与随机值均返回类似 `state=1`、`flag=true` 或 `data=null`，无法证明真实账号/企业枚举。
- 不建议单独提交。

### 4. PMS登录认证
- 测试 `admin`、随机账号、SQLi样式 payload 后，响应表现基本一致。
- 暂未证明验证码绕过、登录绕过或SQL注入。

### 5. 移动端 isec
- `/isec/getServerRandom.do` 返回成功；`/isec/keyAgreement.do` 返回 `9997`。
- 未锁定真实业务basePath或认证绕过前，不构成漏洞。

## 当前最值得继续增强的方向

### PMS注册链验证码发送接口
接口:
- `/pms/ananymous/jyzt/zc/getEmailSession`
- `/pms/ananymous/jyzt/zc/sendEmailYzm`

已验证:
- 公开注册页 `SESSION + _csrf` 即可触发。
- `sendEmailYzm` 对邮箱/手机号验证方式返回 `success`。
- 短时间重复返回 `hassend`，说明进入真实频控/发送流程。
- 无 Cookie/CSRF 对照返回 `403 Invalid CSRF Token`。

但如果平台已忽略，仅有 `success/hassend` 还不够。下一次继续深挖时应增强为以下任一证据:
1. 使用研究员自有邮箱/手机号证明实际收到验证码。
2. 证明同一公开会话可对多个不同邮箱/手机号连续触发。
3. 证明频控仅按单目标限制，换目标可持续触发，形成短信/邮件轰炸或资源滥用。
4. 证明无需登录、无需业务账号即可批量消耗验证码通道资源。

## 报告边界
- 可写: “供应商注册链验证码发送接口未授权调用导致验证码资源被滥用/邮件或短信轰炸”。
- 不要写: “账号接管”“任意用户密码重置”“账号枚举”，除非后续有实际绕过验证码或稳定差异证据。
- 风险级别建议: 有实际收到/批量触发证据时中危；仅有接口返回时容易被忽略。

## 操作偏好
当用户说“漏洞都被忽略了，继续深度挖掘”时:
1. 不要继续汇报低危配置类结果。
2. 不要把多个低危点包装成新增漏洞数量。
3. 优先围绕已接近实质危害的业务链增强证据。
4. 如果没有新的未授权数据、IDOR、认证绕过、RCE/SQLi，直接说明没有新增可提交高价值漏洞。
