# 北京中关村银行(360众测1492期) 实战补充

时间: 2026-05-19
目标: zgcbank.com / corporbankq.zgcbank.com / pms.zgcbank.com / h5.zgcbank.com

## 已确认结论

### 1. corporbankq.zgcbank.com 是典型 SPA fallback
以下路径均返回同一份企业网银 SPA 首页，不能误报为未授权接口:
- `/`
- `/login`
- `/login.json`
- `/cfcalogin/login`
- `/cfcalogin/login.json`
- `/robots.txt`

共同特征:
- `Content-Type: text/html`
- `Content-Length: 1035`
- `<title>中关村银行企业网银</title>`
- 页面引用 `/1.0.18/index.js` 和 CFCA 安全控件 JS

实战判断:
- 不能因为 `GET /login.json` 返回 200 就判定接口存在
- 不能因为 `/robots.txt` 返回 200 就判定敏感文件存在
- 必须同时比较 body、content-type、content-length，确认是否只是前端路由回退

### 2. pms.zgcbank.com/cms 是真实后端应用，不是纯静态占位页
低频验证结果:
- `/cms/` -> 200 HTML
- `/cms/login` -> 404 JSON
- `/cms/api` -> 404 JSON
- `/cms/index.jsp` -> 404 JSON
- `/cms/swagger-ui.html` -> 500 JSON
- `/cms/v2/api-docs` -> 404 JSON
- `/cms/v3/api-docs` -> 404 JSON
- `/cms/swagger-resources` -> 404 JSON
- `/cms/actuator` -> 404 JSON

特征:
- 404/500 是结构化 JSON，而不是网关统一 HTML 404
- `/cms/swagger-ui.html` 返回 500，说明后端确实接收并处理该路由
- 这类行为通常意味着 Spring Boot/Java Web 应用在真实处理请求

实战含义:
- 当 `/swagger-ui.html` 返回 500 而 `/v2/api-docs` `/v3/api-docs` 为 JSON 404 时，优先判断为“真实后端路由异常”，值得继续单点深挖
- 优先继续验证 `/swagger-ui/`、`/doc.html`、`/webjars/`、静态 JS 中的 API 前缀，而不是立刻换目标

### 3. `Access-Control-Allow-Origin: *zgcbank.com` 不能直接报 CORS
现象:
- 普通 GET 响应头出现 `Access-Control-Allow-Origin: *zgcbank.com`
- 但带 `Origin: https://evil.com` 的 OPTIONS 预检请求返回:
  - `403`
  - body: `Invalid CORS request`

判断规则:
- 这种情况不能直接当成“任意 Origin 允许”或“可利用 CORS”
- 必须做预检验证；若预检被拒，则通常不构成可利用的跨域读漏洞

## 当前最优先打法
1. `pms.zgcbank.com/cms`：围绕 swagger 异常和真实后端路由做低频单点验证
2. `corporbankq.zgcbank.com`：离线逆向 `/1.0.18/index.js`，提取真实 API 前缀、CFCA 登录流、下载链接、测试环境线索
3. `h5.zgcbank.com`：不要盲打 `/isec/*`，先从 APK/JS 反推参数和 basePath
